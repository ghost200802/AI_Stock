# 0. 系统总体设计

## 一、设计目标

基于现有 `AI_Stock` 项目，扩展实现股价骨架计算层 C、股价形态判断层 D、显示输出层 E（暂不实现）。最终形成五层架构的股票形态判断系统。

## 二、项目目录架构原则

```
lib/       → 对接外部库的适配层（数据源、数据库等），按职责分子目录
modules/   → 项目自有核心业务逻辑（缠论、蔡森、量价、渲染等）
scripts/   → 基于上述基础模块的定制化脚本
config/    → 全局配置
tests/     → 测试
```

## 五、模块间依赖关系

```
lib/data_source/ (A层)
    ↓ 提供 K线 DataFrame
modules/chanlun/ (C层)
    ↓ 提供笔序列
modules/caisen/ (D层-形态判断)
    ↓ 提供形态识别结果
modules/volume_price/ (D层-量价验证)
    ↓ 提供量价验证结果
modules/analysis/ (D层-综合)
    ↓ 提供交易信号
modules/renderer/ (E层, 暂不实现)

所有层均可通过 lib/db/ 读写 MongoDB 缓存
```

**依赖规则**：
- `lib/` 下的模块不依赖 `modules/`
- `modules/` 可以依赖 `lib/`
- `modules/` 内部各子模块可以互相依赖（如 caisen 依赖 chanlun 的数据结构）
- `scripts/` 依赖 `lib/` 和 `modules/`

### 6.1 缓存策略核心原则
- 所有可持久化数据写入 MongoDB

## 七、lib/db/ 扩展

为支持新增 collection 的索引和查询模式，在 `lib/db/cache_manager.py` 中新增：

```python
class CacheManager:
    def __init__(self, mongodb_manager: MongoDBManager):
        self.db = mongodb_manager

    def upsert_analysis_data(self, collection_name, documents, unique_keys):
        """通用分析数据 upsert，支持自定义唯一键"""

    def find_analysis_cache(self, collection_name, query, sort_key="trade_date"):
        """查询分析缓存数据，返回 DataFrame"""

    def invalidate_cache_after(self, collection_name, ts_code, period, date):
        """使指定日期之后的缓存失效（用于增量更新时清除过期数据）"""
```

## 八、缓存优先读取策略

所有分析模块统一遵循以下缓存策略：

```python
def get_or_compute(ts_code, period, compute_func, collection_name, cache_query):
    cached = cache_manager.find_analysis_cache(collection_name, cache_query)
    if not cached.empty:
        return cached

    result = compute_func(ts_code, period)
    if not result.empty:
        cache_manager.upsert_analysis_data(collection_name, result.to_dict("records"), ...)

    return result
```

增量更新策略：
- 新K线到来时，检查 MongoDB 中该股票+级别的最新缓存日期
- 仅删除/重新计算最新日期之后的分析数据
- 历史已确认的形态和笔数据保持不变
