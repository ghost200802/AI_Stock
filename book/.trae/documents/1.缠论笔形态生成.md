# 1. 缠论笔形态生成

## 一、模块定位

缠论模块位于 **C. 股价骨架计算层**，属于项目核心业务逻辑，放在 `modules/chanlun/` 下。负责从原始K线数据中提取股价的结构化骨架（分型、笔），为上层蔡森形态识别提供输入。同时支持多级别分析，结果通过 `lib/db_manager.py` 缓存到 MongoDB。

**依赖关系**：
- 数据输入：来自 `lib/data_source/` 的 K线 DataFrame
- 缓存读写：通过 `lib/db_manager.py`（`DBManager` 类）操作 MongoDB
- 不依赖其他 modules 子模块

## 二、数据流

```
lib/data_source/data_fetcher.py
    ↓ DataFrame [trade_date, open, high, low, close, volume, ...]
modules/chanlun/include_processor.py
    ↓ List[ProcessedKLine] (经过包含关系处理的K线)
modules/chanlun/fractal_detector.py
    ↓ List[Fractal] (顶分型/底分型序列，经过连续同类型过滤和间距过滤)
modules/chanlun/bi_generator.py
    ↓ List[Bi] (笔序列)
modules/chanlun/__init__.py (compute_bi)
    ↓ DataFrame (分型+笔的叠加视图，用于可视化 Overlay)
```

> **注意**：`multi_level_analyzer.py` 尚未实现，多级别分析功能留作后续开发。

## 三、K线包含关系处理

### 3.1 文件：`modules/chanlun/include_processor.py`

### 3.2 包含关系定义
一根K线的高低点全在另一K线的范围内：
- 包含：`k1.high >= k2.high and k1.low <= k2.low` 或 `k2.high >= k1.high and k2.low <= k1.low`

### 3.3 合并方向判断
- **向上合并**：当当前趋势方向向上时（`result[-2].high > result[-1].high` 或首根K线 `close >= open`）
  - 新K线：`high = max(k1.high, k2.high)`, `low = max(k1.low, k2.low)`
- **向下合并**：当当前趋势方向向下时（`result[-2].high < result[-1].high` 或首根K线 `close < open`）
  - 新K线：`high = min(k1.high, k2.high)`, `low = min(k1.low, k2.low)`

### 3.4 处理流程
```
输入: 原始K线 DataFrame (N行)
输出: List[ProcessedKLine] (M个, M <= N)

算法:
1. 初始化 result = [klines[0]]
2. 遍历每根K线 (从第2根开始):
   a. 判断当前K线与 result[-1] 是否存在包含关系
   b. 若存在包含关系:
      - 通过 _determine_direction() 确定合并方向
      - 按 direction 进行合并，替换 result[-1]
   c. 若不存在包含关系:
      - 加入 result
3. 返回 result
```

### 3.5 数据结构
```python
@dataclass
class ProcessedKLine:
    trade_date: object        # 交易日期
    open: float
    high: float
    low: float
    close: float
    volume: float             # 合并时取两根K线成交量之和
    original_indices: List[int] = field(default_factory=list)  # 被合并的原始K线索引列表
```

### 3.6 缓存策略
- 包含处理结果是中间数据，**不单独持久化**到 MongoDB
- 每次从原始K线数据实时计算（O(n)复杂度，性能足够）
- 若后续发现性能瓶颈，可考虑缓存

## 四、分型识别

### 4.1 文件：`modules/chanlun/fractal_detector.py`

### 4.2 分型定义
基于包含关系处理后的K线序列，取连续三根K线（使用 `>=` / `<=`，允许等值）：
- **顶分型**：中间K线高点为三者最高，低点也为三者最高
  - `k[i-1].high <= k[i].high >= k[i+1].high` 且 `k[i-1].low <= k[i].low >= k[i+1].low`
- **底分型**：中间K线低点为三者最低，高点也为三者最低
  - `k[i-1].high >= k[i].high <= k[i+1].high` 且 `k[i-1].low >= k[i].low <= k[i+1].low`

### 4.3 算法实现
```python
class FractalType(Enum):
    TOP = "top"
    BOTTOM = "bottom"

@dataclass
class Fractal:
    fractal_type: FractalType  # FractalType.TOP / FractalType.BOTTOM
    index: int                 # 处理后K线序列中的索引
    high: float                # 分型最高价
    low: float                 # 分型最低价
    trade_date: object         # 中间K线日期

class FractalDetector:
    def detect(self, processed_klines: List) -> List[Fractal]:
        """
        三步过滤:
        1. _find_fractals()      — 滑动窗口识别原始分型 O(n)
        2. _filter_consecutive_same_type() — 过滤连续同类型分型（保留更极端的）
        3. _filter_by_distance_and_alternation() — 过滤间距不足(<3根K线)的分型
        """
```

### 4.4 连续同类型分型过滤
在 `_find_fractals()` 之后，通过 `_filter_consecutive_same_type()` 处理连续同类型分型：
- 连续顶分型：保留 `high` 更高的
- 连续底分型：保留 `low` 更低的
- 等值时保留第一个

### 4.5 间距与交替过滤
通过 `_filter_by_distance_and_alternation()` 确保分型之间至少间隔 3 根K线（`f.index - last.index >= 3`），并处理间距不足时的冲突解决逻辑。

### 4.6 缓存策略
- 分型数据持久化到 MongoDB，collection 由 `ChanLunCacheManager._make_data_type("fractions", period)` 生成，格式为 `chanlun_fractions_{period}`
- 缓存键：`{ts_code, data_type, trade_date}`
- 通过 `modules/chanlun/cache_manager.py`（`ChanLunCacheManager`）进行缓存读写，底层调用 `lib/db_manager.py`（`DBManager`）
- 增量更新：通过 `invalidate_after()` 删除指定日期之后的分型和笔缓存

## 五、笔生成算法

### 5.1 文件：`modules/chanlun/bi_generator.py`

### 5.2 笔的定义
笔是连接两个相邻顶底分型的价格运动：
- 向上笔：从底分型到顶分型（起始价 = 底分型 `low`，结束价 = 顶分型 `high`）
- 向下笔：从顶分型到底分型（起始价 = 顶分型 `high`，结束价 = 底分型 `low`）

### 5.3 笔的生成条件
1. 分型必须交替出现（顶-底-顶-底...）
2. 顶底分型之间至少有 `min_independent_klines` 根独立K线（默认 1 根）
3. 通过 `_ensure_alternating_direction()` 确保笔方向严格交替

### 5.4 数据结构
```python
class BiDirection(Enum):
    UP = "up"
    DOWN = "down"

@dataclass
class Bi:
    direction: BiDirection    # BiDirection.UP / BiDirection.DOWN
    start_date: object        # 起始日期
    end_date: object          # 结束日期
    start_price: float        # 起始价
    end_price: float          # 结束价
    start_index: int          # 起始分型在处理后K线中的索引
    end_index: int            # 结束分型在处理后K线中的索引
    confirmed: bool = True    # 是否已确认
```

### 5.5 算法实现
```python
class BiGenerator:
    def __init__(self, min_independent_klines: int = 1):
        self.min_independent_klines = min_independent_klines

    def generate(self, fractals: List, processed_klines: List) -> List[Bi]:
        """
        四步生成:
        1. _merge_consecutive_same_type()   — 合并连续同类型分型
        2. _resolve_consecutive_same_type()  — 二次确保分型交替
        3. 遍历分型对，通过 _check_distance() 检查间距，_determine_direction() 判断方向
        4. _ensure_alternating_direction()   — 安全网：确保笔方向严格交替
        """
```

### 5.6 相邻同类型分型处理
当出现连续两个顶分型（或底分型）时，保留更极端的那个：
- 连续顶分型：保留 `high` 更高的
- 连续底分型：保留 `low` 更低的
- 等值时保留第一个

该逻辑在两个阶段执行：
1. `BiGenerator._merge_consecutive_same_type()` — 初次合并
2. `BiGenerator._resolve_consecutive_same_type()` — 二次清理

### 5.7 笔方向交替保证
`_ensure_alternating_direction()` 作为安全网：
- 连续同方向向下笔：保留 `end_price` 更低的
- 连续同方向向上笔：保留 `end_price` 更高的

### 5.8 笔的确认与更新
- 最后一笔始终标记为 `confirmed: False`（未确认笔）
- 其余笔为 `confirmed: True`（已确认笔）

### 5.9 缓存策略
- 笔数据持久化到 MongoDB，collection 由 `ChanLunCacheManager._make_data_type("bis", period)` 生成，格式为 `chanlun_bis_{period}`
- 缓存键：`{ts_code, data_type, trade_date}` 和 `{ts_code, data_type, bi_index}`
- 通过 `modules/chanlun/cache_manager.py`（`ChanLunCacheManager`）进行缓存读写，底层调用 `lib/db_manager.py`（`DBManager`）
